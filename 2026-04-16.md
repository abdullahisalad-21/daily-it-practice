# 🔐 3. Key Concepts Learned

## OAuth 2.0
- Delegated **authorization** framework  
- Uses **access tokens**, not passwords  
- Roles: Resource Owner, Client, Authorization Server, Resource Server  
- Authorization Code Flow: request → consent → code → token → resource access  
- OAuth **does not authenticate** users

## OpenID Connect (OIDC)
- Identity layer built on OAuth  
- Provides **ID tokens (JWT)**  
- Used for login (“Sign in with Google”)  
- OAuth = authorization, OIDC = authentication

## Mobile Security Methods
- Threats: phishing, malicious apps, insecure Wi-Fi, MITM, poor patching  
- Controls: biometrics, PINs, encryption, remote wipe, OS updates, AV, firewalls  
- Policies: AUP, mobile security standards, user training  
- Confidentiality is the primary risk

## Access Control Lists (ACLs)
- Object-level authorization mechanism  
- ACL = list of ACEs (principal + permission + allow/deny + inheritance)  
- **DACL** controls access; **SACL** controls auditing  
- Windows evaluation order: explicit deny → explicit allow → inherited deny → inherited allow  
- Used in NTFS, registry, AD, routers, firewalls

## RADIUS vs TACACS+ vs Accounting vs Auditing
- **RADIUS**: network access, UDP, partial encryption, combined Auth+AuthZ  
- **TACACS+**: device admin, TCP, full encryption, granular command control  
- **Accounting**: session tracking (start/stop, usage, commands)  
- **Auditing**: OS-level security event logging (logons, file access, policy changes)

## AAA Network Flow
1. User sends username/password  
2. NAS sends Access-Request  
3. Server returns Accept/Reject  
4. Accounting-Start  
5. Accounting-Response  
6. User accesses resources  
7. Accounting-Stop  
8. Accounting-Response  
9. Access termination notification

# 🧪 4. Case Study — 2017 Google OAuth Phishing Attack
- Attackers created a malicious OAuth app named **“Google Docs”**  
- Users approved OAuth scopes → attacker received full Gmail access  
- No passwords stolen; OAuth tokens bypassed MFA  
- Worm-like propagation via victim’s contacts  
- Google mitigated by revoking tokens and tightening app verification  
- Demonstrates **authorization abuse**, not authentication failure  
- Reinforces importance of OAuth app whitelisting and scope restrictions

# 🛠️ 5. Technical Workflows Practiced

### OAuth Authorization Code Flow
- Redirect → consent → code → token → API access  
- Token-based access control  
- Separation of identity and resource servers

### ACL Evaluation
- ACE ordering and inheritance  
- Explicit deny precedence  
- DACL vs SACL behavior  
- Practical implications for Windows and network devices

### AAA Protocol Behavior
- RADIUS vs TACACS+ packet handling  
- Accounting start/stop events  
- Device-level vs network-level authorization

# 🧩 6. Sysadmin-Level Insights
- OAuth tokens are a major attack surface; MFA does not protect against malicious app consent  
- ACL misconfiguration is a common cause of privilege escalation  
- TACACS+ is mandatory for secure device administration due to full packet encryption  
- RADIUS remains the standard for Wi-Fi/VPN authentication  
- Accounting and auditing serve different layers: network vs OS  
- Mobile devices require layered controls: technical + policy + user training
